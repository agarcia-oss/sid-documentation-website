Domain Options – Product and/or Resource
The changes is only made to the Characteristic Spec ABE. All other ABEs where this pattern is duplicated are not updated. By promoting the use entities, any entity that does not inherit from the EntitySpecification, or is an entity (RootyEntityType) that does not have a specification, can be related to the use entities.
The choice using this model in the Product and/or Resource domain may be based upon a company’s business model. In the content example presented here, a company may treat each request for content as an order associated with a ProductOffering. Or, a company may treat each request as some form of usage. Or a company may use a hybrid model that combines the treatment of requests.
This model supports all three of these business models, because it can be used for
• Product Spec/Offer Pricing/Charging (as shown in the diagrams in this section)
• Usage Pricing/Charging
• Resource Spec Pricing/Charging.
One other factor needs to be considered when making the choice. The information may have to be duplicated because of application boundaries. A CRM application may need this information as well as Provisioning and Inventory applications.
There are also some application/database implementation considerations
• The use of one set of shared characteristic tables (if the applications are cooperative enough to do this!)
• If the data is only contained in one application then other applications have to request characteristic data from it.
